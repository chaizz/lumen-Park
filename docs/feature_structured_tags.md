# 投稿页标签交互优化设计文档

## 1. 背景与目标 (Background & Goals)

**现状**：
*   首页提供了 **光线 (Lighting)**、**地点 (Location)**、**主题 (Subject)** 三个维度的筛选。
*   投稿页目前仅提供一个通用的标签输入框和一堆混合的推荐标签。
*   **问题**：用户在投稿时感知不到分类维度，导致录入的标签杂乱，后端自动归类（靠猜）准确率低，最终导致首页筛选结果不准确。

**目标**：
1.  **统一维度**：让投稿时的标签选择与首页筛选维度一一对应。
2.  **低交互成本**：避免复杂的下拉菜单或多步骤表单，保持“一键点选”的轻量体验。
3.  **数据准确性**：确保核心标签（如“阴天”、“人像”）能被正确归类。

## 2. 核心交互方案 (Interaction Design)

采用 **“分组快捷标签 (Grouped Quick Tags)”** 方案。

### 2.1 UI 布局变化

将原本杂乱的“推荐标签”区域重构为**三个独立的横向滚动区（或换行区）**，每个区域对应一个分类维度。

**设计草图**：

```text
[ 场景标签 ]
(已选标签展示区: [阴天 x] [咖啡厅 x])
--------------------------------------------------
✨ 光线:  [ 阴天 ] [ 晴天 ] [ 室内 ] [ 夜景 ] [ 黄金时刻 ]
📍 地点:  [ 咖啡厅 ] [ 街道 ] [ 海边 ] [ 公园 ] [ 居家 ]
📷 主题:  [ 人像 ] [ 静物 ] [ 扫街 ] [ 建筑 ] [ 胶片感 ]
--------------------------------------------------
[ + 自定义标签输入 ]
```

### 2.2 交互逻辑

1.  **点选即中**：用户点击“阴天”，该标签立即变为**选中态**（高亮色），并加入到提交列表。再次点击取消。
2.  **视觉引导**：每个维度的标签前加一个小图标（✨/📍/📷），潜移默化地告诉用户“这是光线”、“这是地点”。
3.  **自定义补充**：保留自定义输入框，用于补充长尾标签（如具体的相机型号、特殊的滤镜名）。自定义标签默认归类为 `other`，不强求用户手动分类（降低负担）。

## 3. 数据与后端逻辑 (Data & Backend)

### 3.1 预设标签库 (Presetted Tags)

后端需要初始化（Seed）一批带有正确 `type` 的标签数据。

*   **Lighting**: `sunny`, `overcast`, `indoor`, `night`, `flash`, `golden_hour`
*   **Location**: `street`, `cafe`, `seaside`, `park`, `home`, `mountain`, `urban`
*   **Subject**: `portrait`, `still_life`, `landscape`, `architecture`, `cat`, `dog`, `food`

### 3.2 提交逻辑

前端在提交时，依然只提交 `tags: string[]` 数组。

*   **前端**：无需改动数据结构，只需改变 UI 展示方式。
*   **后端**：
    *   当收到 `tags=["阴天", "咖啡厅"]` 时。
    *   数据库中查询 "阴天" -> 发现已存在且 type="lighting" -> 关联之。
    *   如果用户输入了新词 "赛博朋克" -> 数据库不存在 -> 创建新标签，type="other"。

*这种方式将分类的复杂性转移到了“预设数据的准备”上，用户端的操作依然只是“点选”。*

## 4. 开发计划 (Action Plan)

1.  **数据准备**：
    *   编写一个 Python 脚本或 SQL，预置上述三类常用标签，并确保其 `type` 字段正确。
2.  **前端改造 (`Submit.vue`)**：
    *   定义三个常量数组，分别存储三类预设标签。
    *   重构模板，将推荐标签区拆分为三行。
    *   优化选中样式，使其与首页的筛选风格保持一致（Sage Green）。

## 5. 预期效果

用户在上传照片时，看到“光线”一行，会自然地去选一个；看到“地点”，也会顺手选一个。这样既完成了分类信息的收集，又没有增加额外的思考负担（比如“这个标签该算哪类？”）。
